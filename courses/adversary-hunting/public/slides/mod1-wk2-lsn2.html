What is Static Analysis?

  • Definition: The process of analyzing a malware sample without executing it. We are dissecting the file itself, not observing its behavior.

The Analogy

It's like reading the blueprints of a building to understand its layout, purpose, and materials, instead of walking through it.

Why start here?

  • SAFETY: It is inherently safe. The malicious code is never run.
  • SPEED: It's a fast way to perform initial triage and determine if a file is worth a deeper, more time-consuming analysis.
  • IOCs: It can quickly reveal hard-coded indicators like IP addresses, domains, filenames, and mutexes.

The Triage Process: First 15 Minutes

When you get a new potential malware sample, your goal is to answer these questions quickly:

1

Is it packed or obfuscated?
(Look for high entropy, few readable strings).

2

What can it do?
(Imports/Libraries give clues, e.g., URLDownloadToFile means it can download things).

3

Are there any easy IOCs?
(Search for strings like IPs, domains, user agents).

4

Is it signed?
(A valid signature from a trusted publisher is less likely to be malware, but not impossible).

Our toolkit for this triage is simple but effective.

The Toolkit: Hashes, Strings, Pestudio

1. File Hashing

MD5, SHA256

The first step is always to get the hash. This is the file's unique fingerprint.

Use the hash to check against threat intelligence sources like VirusTotal. Has someone else already done the analysis for you?

2. Strings

Command Line Utility

A simple utility that extracts any ASCII and Unicode text sequences from a binary file.

It's a noisy but often fruitful way to find domains, IPs, file paths, registry keys, and error messages.

3. Pestudio

All-in-one GUI

A powerful, all-in-one tool for static analysis of Windows executables.

It consolidates information about headers, libraries (imports), strings, resources, and provides a threat assessment. Our go-to tool for initial triage.

Ghidra: Reading the Blueprints

After the initial triage, we move to a more powerful tool. Ghidra allows us to see the fundamental logic of the program.

Symbol Tree:

Shows all the functions in the program. We often look for function names that hint at certain behaviors (e.g., createC2Beacon, encryptFile).

Decompiler Window:

This is where the magic happens. It translates the raw assembly code into readable, C-like pseudocode.

Our Goal Today

Not to become expert reverse engineers, but to learn how to navigate the interface, find the main or entry function, and get a high-level sense of the program's flow.
